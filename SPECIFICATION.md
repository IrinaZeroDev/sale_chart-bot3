# Согласованная спецификация

## Исходные допущения

- Реальной CRM, прайса и API заказчик не предоставляет.
- Интеграция демонстрируется на реалистичной in-memory системе с REST API.
- Основной канал — Telegram: он поддерживает webhook и не требует создания группы или канала. Без токена интеграция работает как mock.
- Бот не сообщает неподтверждённые цены, сроки, статусы и условия.

## Сценарий

Приветствие: «Здравствуйте. Чем могу помочь: информация о товарах, статус заказа или вопросы поддержки?»

1. **Информация о товарах.** Бот находит демо-продукты, сообщает цену и ориентировочный срок из каталога.
2. **Статус заказа.** Бот запрашивает номер формата `ORD-1001`, проверяет его в тестовой системе и возвращает только найденные данные.
3. **Поддержка.** Бот отвечает по согласованным темам: доставка, сроки, оплата и возврат. При отсутствии точного ответа предлагает менеджера.
4. **Передача менеджеру.** Бот сообщает о передаче обращения. Телефон или email сохраняется только после явного согласия клиента.

## Демо-интеграции и отчётность

- каталог: `GET /api/v1/demo/products`;
- заказ: `GET /api/v1/demo/orders/{order_id}`;
- созданные лиды: `GET /api/v1/demo/leads`;
- метрики: `GET /api/v1/metrics` — диалоги, сообщения, лиды, конверсия, передачи менеджеру и темы;
- чат: `POST /api/v1/chat`;
- Telegram: `POST /webhook/telegram` в Node.js-шлюзе.

## Критерии приёмки

- меню сформулировано профессионально и лаконично;
- цены берутся из каталога, а статусы — из базы заказов;
- неизвестный заказ не получает выдуманный статус;
- FAQ покрывает цены, доставку и сроки;
- неизвестный вопрос допускает передачу менеджеру;
- контакт без согласия не сохраняется;
- после согласия создаётся лид с уникальным ID;
- REST endpoints защищены внутренним API key;
- GigaChat, ProxyAPI и mock выбираются настройкой окружения.
